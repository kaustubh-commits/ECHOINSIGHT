# ADR-0003: Composing SegmentDSP from Domain-Specific Value Objects

**Status:** Accepted  
**Date:** 2026-07-21

---

## Context

Every structural segment in EchoInsight contains a collection of deterministic Digital Signal Processing (DSP) measurements describing its acoustic characteristics.

These measurements span multiple conceptual domains, including:

- Rhythm
- Harmony
- Timbre

Although they are ultimately part of the same segment, they represent different aspects of musical analysis and evolve independently as the project grows.

The architectural question was:

> Should all DSP measurements exist as fields on a single class, or should they be grouped into domain-specific value objects?

This decision affects readability, validation, extensibility, comparison algorithms, and the long-term maintainability of the domain model.

---

## Decision Drivers

The decision prioritized:

- Separation of concerns
- Domain clarity
- Maintainability
- Type safety
- Extensibility
- Alignment with musical concepts

---

## Decision

`SegmentDSP` is implemented as a composition of domain-specific value objects.

```text
SegmentDSP
├── SegmentRhythm
├── SegmentHarmony
└── SegmentTimbre
```

Each sub-object owns a cohesive group of related measurements and is responsible for validating its own invariants.

Consumers access measurements through their conceptual category.

```python
segment.dsp.rhythm.tempo

segment.dsp.harmony.key

segment.dsp.timbre.rms_mean
```

This organization mirrors the structure of the musical domain rather than treating DSP measurements as a flat collection of unrelated values.

The aggregate object (`SegmentDSP`) exists primarily as a container that groups these independent analytical dimensions into a single representation of a segment.

---

## Alternatives Considered

### Flat Structure

Store every DSP measurement directly on `SegmentDSP`.

```python
SegmentDSP(
    tempo=...,
    tempo_confidence=...,
    chroma_mean=...,
    mfcc_mean=...,
    ...
)
```

**Rejected because**

A flat structure provides no meaningful organization.

As the feature set grows, the class naturally becomes a dumping ground for unrelated measurements, making it increasingly difficult to navigate, validate, and extend.

Constructors become harder to read, validation logic becomes centralized into one large method, and developers lose clear conceptual boundaries when introducing new features.

---

### Dictionary-Based Representation

Represent DSP measurements as a generic mapping.

```python
{
    "tempo": ...,
    "rms": ...,
    "centroid": ...
}
```

**Rejected because**

A dictionary removes the domain model entirely.

Measurements become identified by string keys instead of typed attributes.

Compile-time checking, IDE support, discoverability, and localized validation are all lost.

The burden of correctness shifts from the domain model to every consumer.

---

## Consequences

### Advantages

- Groups related measurements into cohesive domain concepts.
- Keeps validation localized to each analytical category.
- Produces a cleaner and more navigable API.
- Naturally supports comparison algorithms that operate on rhythm, harmony, or timbre independently.
- Allows each analytical category to evolve without increasing the complexity of unrelated components.

### Disadvantages

- Attribute access becomes slightly deeper (`segment.dsp.rhythm.tempo`).
- Additional classes introduce a small increase in structural complexity.
- Nested serialization produces a slightly larger JSON representation.

### Tradeoffs

The additional object hierarchy is intentional.

EchoInsight prioritizes conceptual clarity over minimizing attribute depth.

The resulting model is easier to understand, easier to maintain, and better aligned with the problem domain than a single monolithic class.

---

## Future Considerations

As new categories of musical analysis emerge, they should be evaluated independently rather than appended to existing structures.

If a new family of measurements represents a distinct analytical concept—for example spatial features, production characteristics, or expressive performance metrics—it should generally become its own value object owned by `SegmentDSP`.

The guiding principle is:

> New analytical concepts should introduce new domain objects rather than expanding existing ones beyond their natural responsibilities.
