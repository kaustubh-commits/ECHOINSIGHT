# ADR-0002: Immutable Domain Model

**Status:** Accepted

---

## Context

EchoInsight's domain objects represent measurements of a real thing — a song. A song's extracted features do not change after analysis. If the extraction pipeline is run twice with the same configuration, it should produce identical output. The domain model should reflect this invariant: once a SongDNA is constructed, it should never mutate.

The question was: **Should domain objects be mutable or immutable?**

---

## Decision

**All domain objects are immutable frozen dataclasses.**

```python
@dataclasses.dataclass(frozen=True)
class SegmentTiming:
    start_time: float
    end_time: float
    ...
```

Every domain type uses `frozen=True`. There are no setters, no mutable methods, and no lazy-initialized properties. If a value needs to change, a new object must be constructed.

Rationale:

- **Correctness by construction.** A frozen dataclass cannot be accidentally modified after creation. This eliminates a class of bugs where a reference to a domain object is shared between consumers and one consumer mutates it, affecting the other.
- **Hashable.** Frozen dataclasses are hashable. This makes them usable as dictionary keys and set members, which is important for caching and memoization in the comparison engine.
- **Serialization-friendly.** `dataclasses.asdict()` produces a clean dict that can be serialized to JSON without custom encoders. Mutable objects would require serialization logic to snapshot state at a point in time.
- **Thread safety.** Immutable objects are safe to share across threads without locking. This matters for future batch processing where multiple songs may be analyzed concurrently.
- **Intent signalling.** A frozen dataclass communicates to future engineers: "this object represents a fact, not a work in progress." It discourages patterns where domain objects are gradually populated after construction.

---

## Alternatives Considered

### Mutable dataclasses

Standard Python dataclasses (non-frozen). Fields are modifiable.

**Rejected because:** Mutable objects introduce temporal coupling — the validity of an object depends on when you access it. A consumer that reads `segment.timing.duration` at two different points may get different values. This makes reasoning about program state significantly harder, especially in a system where domain objects are passed between services.

### Pydantic models

Pydantic provides validation and serialization with mutable (by default) or immutable (via `frozen=True`) models.

**Rejected because:** Pydantic is an external dependency. EchoInsight's domain layer (`src.core/`) must import only the Python standard library. Introducing Pydantic would create a dependency from the domain layer to an external package, violating the layer architecture. Pydantic also adds runtime overhead for validation that we didn't need — the domain model performs only lightweight structural validation in `__post_init__`.

### NamedTuple

Python's `NamedTuple` provides immutable, lightweight data containers.

**Rejected because:** NamedTuples do not support `__post_init__` validation, default factories, or inheritance. They are also less readable than dataclasses for complex nested structures. For simple value objects like `BoundaryConfidence`, NamedTuple would work, but the inconsistency of mixing NamedTuples and dataclasses in the same module would harm readability.

---

## Consequences

**Advantages:**
- Eliminates mutation-related bugs
- Thread-safe by design
- Clean JSON serialization via `dataclasses.asdict()`
- Clear contract: construct once, read forever

**Disadvantages:**
- Performance penalty for "updates": changing a single field requires constructing a new object (using `dataclasses.replace()`)
- Nested immutability requires care: all fields must themselves be immutable types (tuple, frozen dataclass, etc.)
- `__post_init__` validation requires `object.__setattr__` for computed fields (like `SegmentTiming.duration`), which is a workaround that could confuse readers

**Tradeoff accepted:** The performance cost of constructing new objects is negligible for domain models (typically 10-50 objects per song). The correctness benefits far outweigh the construction overhead.

---

## Future Notes

If performance profiling ever shows object construction as a bottleneck, the hot paths can use `__slots__` (already used on `Estimate`) or manual `__init__` methods. But this should be driven by profiling data, not premature optimization.
