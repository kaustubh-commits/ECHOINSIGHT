# ADR-0006: Comparison Contracts

**Status:** Accepted

---

## Context

EchoInsight's comparison engine measures similarity between songs and segments across multiple dimensions (tempo, timbre, harmony, energy, structural flow). The output of a comparison is a structured result containing scores, explanations, and specific differences.

The question was: **Where should comparison result types live ‚Äî in the domain layer (`src.core`) or in the analysis layer (`src.analysis`)?**

---

## Decision

**Comparison result types live in `src.analysis.comparison.contracts`, NOT in `src.core`.**

The domain layer (`SongDNA`, `SegmentDNA`, `StructureDNA`) has no knowledge of comparison. The comparison engine consumes domain types and produces its own result types.

This separation is enforced by the layer architecture:

```
src.core/          ‚Üí SongDNA, SegmentDNA (no comparison knowledge)
src.analysis/      ‚Üí ComparisonEngine, ComparisonResult (consumes domain types)
```

The comparison result types are:

- `ComparisonDimension` ‚Äî score for a single dimension (e.g., "tempo: 0.92")
- `ComparisonDifference` ‚Äî a specific measurable difference between two entities
- `ComparisonResult` ‚Äî full comparison of two entities, with overall score, dimension breakdown, and explanations
- `SegmentMatch` ‚Äî a single segment match from a similarity search
- `SearchResults` ‚Äî collection of ranked matches

---

## Alternatives Considered

### Put comparison types in the domain layer

Define `ComparisonResult` alongside `SongDNA` in `src.core`.

**Rejected because:**
- The domain layer would need to know about comparison ‚Äî a cross-cutting concern
- Adding new comparison dimensions (e.g., structural flow) would require modifying domain types
- The domain layer imports only stdlib; comparison types would need to import domain types (acceptable) but domain types would need to import nothing (violated if comparison types live there)
- Every future algorithm that produces results would need to put its result types in the domain layer, creating a dumping ground

### Embed comparison results in SongDNA

Each `SongDNA` stores pre-computed comparison results with similar songs.

**Rejected because:** Comparison results are a function of two songs, not a property of a single song. If song A's comparison to song B changes (because song B is re-analyzed), song A's stored results would be stale. Comparison is a service, not a domain attribute.

### Unstructured dict output

The comparison engine returns plain dicts rather than typed objects.

**Rejected because:** Dicts provide no type safety, no validation, and no discoverability. A consumer must know the dict keys to use the result. IDEs cannot provide autocompletion. Validation is ad-hoc. This pattern was used in the v1 comparison engine and led to brittle code.

---

## Consequences

**Advantages:**
- Clear separation of concerns: domain types describe what a song IS; comparison types describe how songs RELATE
- The domain layer remains pure ‚Äî no dependency on comparison algorithms
- Comparison types can evolve independently of the domain schema
- Every comparison result has an `evidence` field ‚Äî explainability is baked into the contract, not added as an afterthought

**Disadvantages:**
- More types to maintain (5 comparison-specific types vs inlining everything into dicts)
- Consumers must import from `src.analysis.comparison.contracts` instead of finding everything in `src.core`
- Comparison types duplicate some structure from domain types (e.g., `SegmentMatch` contains a `SegmentDNA`)

**Tradeoff accepted:** The duplication is minimal and the separation prevents the domain layer from becoming a dumping ground for every algorithm's output format.

---

## Future Notes

If the recommendation engine is built, it should follow the same pattern: result types live in `src.analysis.recommendation.contracts`, not in the domain layer. The `song_id` identifier is the linking key across all result types.
