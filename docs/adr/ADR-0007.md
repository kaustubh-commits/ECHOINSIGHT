# ADR-0007: Versioning Domain Schema Evolution

**Status:** Accepted  
**Date:** 2026-07-21

---

## Context

EchoInsight persists its domain model as serialized data.

As the project evolves, domain objects will inevitably change.

Examples include:

- introducing new fields
- deprecating existing fields
- refining analytical models
- restructuring aggregates
- correcting earlier design decisions

Previously generated data must remain readable by newer versions of the software.

The architectural question was:

> How should domain schema evolution be managed without tightly coupling persisted data to a specific software release?

This decision affects backward compatibility, migration strategy, and the long-term evolution of the domain model.

---

## Decision Drivers

The decision prioritized:

- Backward compatibility
- Explicit evolution
- Independent aggregate lifecycles
- Predictable migrations
- Maintainability

---

## Decision

Every aggregate root carries its own explicit `schema_version`.

```python
@dataclass(frozen=True)
class SongDNA:
    schema_version: str = "2.0.0"
```

Schema versions identify the structure of persisted domain objects rather than the version of the application itself.

Each aggregate evolves independently.

For example:

- `SongDNA`
- `StructureDNA`
- `SegmentDNA`

may progress through different version histories as their responsibilities change.

Schema versions follow Semantic Versioning principles.

- **Major** — incompatible structural change
- **Minor** — backward-compatible structural extension
- **Patch** — non-structural correction

Persisted data is migrated when loaded.

Migration functions transform serialized representations from one schema version to the next until the current version is reached.

The domain model therefore interacts only with the latest schema.

---

## Alternatives Considered

### Rewrite Persisted Data

Whenever the schema changes, rewrite every stored file into the newest format.

**Rejected because**

This tightly couples schema evolution to operational workflows.

Large datasets become expensive to update, migration failures risk data corruption, and every schema revision requires users to modify existing data before it can be read.

---

### Global Schema Version

Maintain one version number for the entire persistence model.

**Rejected because**

Different aggregates evolve at different rates.

A structural change to one aggregate should not require unrelated aggregates to appear version-incompatible.

Independent version histories provide better isolation and reduce unnecessary migrations.

---

### External Schema Definitions

Maintain versioned schemas separately using technologies such as JSON Schema.

**Rejected because**

EchoInsight's domain model is defined directly in Python.

Maintaining duplicate schema definitions introduces additional maintenance burden and increases the likelihood of divergence between the implementation and the persisted format.

---

## Consequences

### Advantages

- Older persisted data remains readable by newer software.
- Schema evolution becomes explicit and traceable.
- Aggregate lifecycles remain independent.
- Migration logic is isolated from the domain model itself.
- Breaking changes become intentional rather than accidental.

### Disadvantages

- Migration code must be maintained while legacy versions remain supported.
- Loading older data may require additional transformation work.
- Engineers must update schema versions whenever structural changes are introduced.

### Tradeoffs

EchoInsight accepts a modest maintenance cost for migration logic in exchange for long-term compatibility.

Persisted analytical data represents user investment.

Preserving that investment is considered more valuable than simplifying schema evolution by requiring manual data conversion.

---

## Future Considerations

Migration support should be maintained only for versions that remain within the project's compatibility policy.

As the schema stabilizes, obsolete migration paths may be retired according to that policy.

Future persistence technologies should continue to follow the same principle:

> Persisted data should evolve through explicit versioned migrations rather than implicit assumptions about its structure.
