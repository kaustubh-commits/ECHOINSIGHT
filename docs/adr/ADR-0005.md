# ADR-0005: Validation Strategy

**Status:** Accepted

---

## Context

Domain objects in EchoInsight must enforce invariants — rules that describe what constitutes a valid object. For example: a segment's start time must be before its end time, a tempo must be positive, a chroma vector must have exactly 12 elements, and the sum of segment durations must equal the song's total duration.

The question was: **Where should validation live and how expensive should it be?**

---

## Decision

**Validation is split into two tiers:**

### Tier 1: Structural validation in `__post_init__` (always executed)

Every frozen dataclass validates its own fields at construction time. These checks are:

- **Cheap** — O(1) or O(n) for small n (e.g., checking tuple lengths)
- **Local** — only uses fields on the object being constructed, never accesses external state
- **Deterministic** — same input always produces same validation result

Examples of structural validation:

| Class | Validation |
|-------|------------|
| `SegmentTiming` | `start_time >= 0`, `end_time > start_time` |
| `SegmentRhythm` | `tempo > 0`, `0 <= confidence <= 1` |
| `SegmentHarmony` | `len(chroma_mean) == 12` when present |
| `SegmentTimbre` | `len(mfcc_mean) == 13` when present, `0 <= zcr <= 1` |
| `SegmentContext` | `index >= 0`, `0 <= novelty_score <= 1` |
| `BoundaryConfidence` | `time >= 0`, `0 <= confidence <= 1`, non-empty `method` |
| `SongMetadata` | `duration > 0`, `sample_rate > 0`, `channels in {1, 2}` |
| `SongSummary` | `0 <= tempo_confidence <= 1`, `len(mfcc_mean) == 13` when present |
| `AlternativeSegmentation` | Non-empty segments, temporally contiguous |

### Tier 2: Cross-field invariant validation in `__post_init__` (always executed)

These validations involve relationships between fields or between an object and its children. They are still executed at construction time but are more expensive because they involve iteration over collections.

Examples of cross-field validation:

| Class | Validation | Cost |
|-------|------------|------|
| `StructureDNA` | Segment durations sum to `total_duration` | O(n) for n segments |
| `StructureDNA` | Segment indices match their position | O(n) |
| `StructureDNA` | Boundary count equals n-1 | O(1) |
| `StructureDNA` | Segments are temporally contiguous | O(n) |
| `SongDNA` | Structure duration matches metadata duration | O(1) |
| `SongDNA` | Every segment's song_id matches metadata song_id | O(n) |
| `AlternativeSegmentation` | Segments are temporally contiguous | O(n) |

The rule is:

> **Tier 1 always fires. Tier 2 always fires. Both are executed at construction time and are non-negotiable.**

No lazy validation, no deferred validation, no "validation service." The object guarantees its own validity.

---

## Alternatives Considered

### Separate validator classes

Move all validation out of domain objects and into dedicated `SongDNAValidator`, `StructureDNAValidator`, etc.

**Rejected because:** Validator classes create a split responsibility — the domain object defines the data, but another class determines whether it's valid. This means:
- A consumer can construct an invalid object and pass it around
- The validator must have knowledge of the object's internals (breaking encapsulation)
- Engineers must remember to call validation explicitly

This pattern works for systems where object construction and validation are separate concerns (e.g., form validation in web apps). For EchoInsight, where domain objects are constructed once and never mutated, validation in the constructor is the correct approach.

### Validation via Python properties

Use `@property` setters to validate values as they're set, rather than in `__post_init__`.

**Not applicable:** Frozen dataclasses don't have setters. The object's state is set once at construction time, making `__post_init__` the natural validation hook.

### Validation in external schema (JSON Schema, Pydantic)

Define validation rules in a separate schema file.

**Rejected because:** The validation rules would be decoupled from the Python types that use them. An engineer modifying a dataclass would need to remember to update the external schema. This creates a source of drift where the schema says one thing and the code does another.

---

## Consequences

**Advantages:**
- Objects guarantee their own validity — a constructed `StructureDNA` is always structurally sound
- No separate validation code path to maintain
- Fail-fast: invalid data is caught at the point of construction, not later when it causes a hard-to-diagnose bug
- Tests can verify validation by constructing invalid inputs and asserting exceptions

**Disadvantages:**
- Construction is slower for objects with many children (StructureDNA's O(n) validation for n segments)
- Cannot partially construct an object and fill in fields later (frozen dataclass constraint, not specific to validation)
- Error messages from `__post_init__` are limited to what the constructor can express — cannot reference external context (e.g., "segment 3 overlaps with segment 4 from song X")

**Performance note:** For typical songs (4-12 segments), validation completes in microseconds. The O(n) cost is not a practical concern. If a song ever has thousands of segments, validation might take milliseconds — still negligible.

---

## Future Notes

If the system ever supports user-editable boundaries (drag-and-drop segmentation in a UI), the validation will need to handle partial updates. In that case, a separate "validation report" method that collects all errors (rather than raising on the first) would be useful. This is not needed now because extraction is the only path that produces segments.
